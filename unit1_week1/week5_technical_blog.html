Ruby is so Classy!

Let's build a new Ruby app with classes!

I love traveling, but I HATE being a tourist. I always wish that I knew some locals that could show me around, or tell me the cool places to go see, or just guide me toward things locals like to do. I prefer to experience places for what they are, instead of hitting up all the usual tourist traps. From this frustration, my little baby was born: I call her Lokalizr.

Lokalizr is supposed to be an app that bridges the gap between travel and comfort. I don't speak Japanese, but I want to go to Japan! I don't know anyone in Japan :(. You know what would be cool? A local guide! That spoke some English! He/she could sit down with me and explain some things, maybe show me around, maybe we'd have lunch, etc. I would be the tourist, but I'd be touring the guide's city through the eyes of a local. Or even better, I want to move to Japan! Oh hell, I can search online forever for tips, but wouldn't it be nice talking to someone that could answer my personal questions? Maybe talking to a few people?

That's the basic idea of the app. So how do we build this thing? The first thing we should do is talk about objects and classes.

WHAT IS AN OBJECT?
"An object is an entity that serves as a container for data and also controls access to the data. Associated with an object is a set of attributes, which are essentially no more than variables belonging to that object. Also associated with an object is a set of functions that provide an interface to the functionality of the object, called methods." - Hal Fulton

Remember how almost everything in Ruby is an object? Think of an object as a jar (or container, as mentioned above). The jar stores a bunch of things inside it, and we can do certain things with the jar as well. So in our app, we'd need to build some objects that would store data, and that we can manipulate. What would those objects be? Classes, of course!

WHAT IS A CLASS?
"A class is just a way of organizing and producing objects with similar attributes and methods." - codeacademy.com

What does this mean, though? It means that a class is an object, and that you can have instances of that class (which are also objects). For example:

class Dog
end

fido = Dog.new("golden retriever")
jumpy = Dog.new("beagle")
fluffy = Dog.new("yorkshire terrier")

Dog would be the entire class, and fido, jumpy, and fluffy would be INSTANCES of that class, but they are all objects.

This leads up into discussing variables, because we need to understand how to use these new classes and objects we're going to creat.

"Ruby provides four types of variables: Local Variables: Local variables are the variables that are defined in a method. Local variables are not available outside the method. You will see more details about method in subsequent chapter. Local variables begin with a lowercase letter or _.

Instance Variables: Instance variables are available across methods for any particular instance or object. That means that instance variables change from object to object. Instance variables are preceded by the at sign (@) followed by the variable name.

Class Variables: Class variables are available across different objects. A class variable belongs to the class and is a characteristic of a class. They are preceded by the sign @@ and are followed by the variable name.

Global Variables: Class variables are not available across classes. If you want to have a single variable, which is available across classes, you need to define a global variable. The global variables are always preceded by the dollar sign ($)." - tutorialspoint.com

Things an object knows about itself are called instance variables. They represent an object's state (the data - for example, the quantity and the product id), and can have unique values for each object of that type. That's why the above explanation of the 4 different types of variables is important.

The next important element to understand are methods, which are things an object can do. There are class methods and there are instance methods. Sometimes a class needs to provide methods that work without being tied to any particular object, and that's why class methods come in handy. Things like deleting and even creating are class-level methods, because they need to work on every instance of that class. Instance methods are called on particular instances of a class, like called fido.bark. If you want ALL your dogs to bark, then build it like this: self.bark. If you want only some dogs to bark, and the rest to sit quietly, then make define your method like so: bark. The "self" keyword means it will work on every single object that you create from that class, which is not always desired.

Now let's talk inheritance (not money, sorry).

"Some object-oriented languages (notably C++) support multiple inheritance, where a class can have more than one immediate parent, inheriting functionality from each. Although powerful, this technique can be dangerous, as the inheritance hierarchy can become ambiguous. Other languages, such as Java, support single inheritance. Here, a class can have only one immediate parent. Although cleaner (and easier to implement), single inheritance also has drawbacks—in the real world things often inherit attributes from multiple sources (a ball is both a bouncing thing and a spherical thing, for example). Ruby offers an interesting and powerful compromise, giving you the simplicity of single inheritance and the power of multiple inheritance. A Ruby class can have only one direct parent, and so Ruby is a single-inheritance language. However, Ruby classes can include the functionality of any number of mixins (a mixin is like a partial class definition). This provides a controlled multiple-inheritance-like capability with none of the drawbacks. We'll explore mixins more in the section “Mixins” in the chapter on modules." - phrogz.net
